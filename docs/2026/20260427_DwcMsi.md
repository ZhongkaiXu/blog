作者: 徐仲锴 (xuzhongkai25e@ict.ac.cn)

# PCI-MSI 和 ARCH-MSI

MSI中断是供PCI设备使用的一种中断方式，但不是所有中断控制器都支持MSI，在ARM架构中只有带有ITS子系统的GIC才支持MSI，在RISC-V中的PLIC是不支持MSI的。

在一些板子上，比如iMX8MP，不带有ITS子系统，就会限制PCI设备的使用，比如无法开启SRIOV。这种情况下，一般会启用DW PCIe控制器自带的MSI机制。

DesignWare PCIe控制器提供了 `PCI-MSI` 机制，在架构的中断控制器不支持MSI的情况下可以选择启用，此时 `cat /proc/interrupts` 会看到PCI设备使用的中断为 `PCI-MSI` 。

## PCI-MSI的特点

在一般的中断控制器中，MSI可以当作一个独立的中断来看待（也就是说处理方式和线中断类似），即，给dev0分配了8192这个中断号，给dev1分配了8193这个中断号，后续根据触发的中断号是8192还是8193就能匹配到不同设备的处理函数。

而在PCI-MSI中，MSI中断在物理上只有一个中断号（例如为177），而分配给设备的是bit。例如，给dev0分配bit0，给dev1分配bit1，后续不管是哪个设备触发中断，被拉高的物理中断号都是177，而软件要去查看记录中断pending状态的寄存器，看哪个bit被拉高，才能判断出需要匹配哪个设备的处理函数。

后续的内容会细讲该机制。

# Doorbell触发地址生成

在Arch的中断控制器支持的MSI机制中，如ARM GIC-ITS子系统中MSI触发的固定寄存器为 `GITS_TRANSLATER` (硬件确定时该地址就确定)，PCIe设备发出的MEM Write TLP报文的目标地址与这个地址匹配时，表明设备要发出中断。

而PCI-MSI的doorbell触发地址，是软件在初始化时动态生成的，同一个板子每次启动时分配的地址也可能不同。将该地址写入DBI的 `PCIE_MSI_ADDR_LO` 和 `PCIE_MSI_ADDR_HI` 寄存器作为MSI doorbell。

Linux中的源码：

```c
#define PCIE_MSI_ADDR_LO		0x820
#define PCIE_MSI_ADDR_HI		0x824

void dw_pcie_msi_init(struct pcie_port *pp)
{
	struct dw_pcie *pci = to_dw_pcie_from_pp(pp);
	struct device *dev = pci->dev;
	u64 msi_target;

	pp->msi_page = alloc_page(GFP_KERNEL);
	pp->msi_data =
		dma_map_page(dev, pp->msi_page, 0, PAGE_SIZE, DMA_FROM_DEVICE);
	if (dma_mapping_error(dev, pp->msi_data)) {
		dev_err(dev, "Failed to map MSI data\n");
		__free_page(pp->msi_page);
		pp->msi_page = NULL;
		return;
	}
	msi_target = (u64)pp->msi_data;

	/* Program the msi_data */
	dw_pcie_wr_own_conf(pp, PCIE_MSI_ADDR_LO, 4, lower_32_bits(msi_target));
	dw_pcie_wr_own_conf(pp, PCIE_MSI_ADDR_HI, 4, upper_32_bits(msi_target));
}
EXPORT_SYMBOL_GPL(dw_pcie_msi_init);
```

# 中断的使能、屏蔽和pending

一个DW控制器最多支持256个中断，每32个中断为一组，共用一组32bit寄存器，寄存器中的每个bit代表一个中断的状态。

```c
#define PCIE_MSI_INTR0_ENABLE		0x828
#define PCIE_MSI_INTR0_MASK		0x82C
#define PCIE_MSI_INTR0_STATUS		0x830
```

- ENABLE：使能中断，中断来了会触发。
- MASK：屏蔽中断，中断来了会在STATUS寄存器里面记录但不会触发，该寄存器很少使用。
- STATUS：中断pending表，某个bit为1代表中断产生，向该bit写回1代表deactivated。

tips：在DW的驱动实现模板中有判断最大中断数的ops，在不实现该ops的情况下默认该控制器只支持32个中断，即只考虑 `INTR0_*` 这一组寄存器。

# 设备的MSI msg的填充

当PCI设备的MSI依赖不同的中断控制器时，设备的MSI msg的填充的信息是不同的，如在LoongArch中会直接填入物理中断号，而在GIC-ITS中会填入事件id，后经过ITS的翻译表将事件id转为中断号发往目标CPU。

在PCI-MSI机制中，设备的MSI msg的内容即为该entry分配到的中断位，例如填入1，则代表该entry发出的MSI中断，会使 `PCI_MSI_INTR0_STATUS` 中的bit1拉高。

在给设备配置msg时，Linux首先通过 `dw_pcie_irq_domain_alloc` 函数找到一个空闲的PCI-MSI bit，再将该bit和一个空闲的virq联系起来。

```c
static int dw_pcie_irq_domain_alloc(struct irq_domain *domain,
				    unsigned int virq, unsigned int nr_irqs,
				    void *args)
{
	printk("dwc-msi: virq-base: %u, nr_irqs: %u\n", virq, nr_irqs);
	struct pcie_port *pp = domain->host_data;
	unsigned long flags;
	u32 i;
	int bit;

	raw_spin_lock_irqsave(&pp->lock, flags);

	bit = bitmap_find_free_region(pp->msi_irq_in_use, pp->num_vectors,
				      order_base_2(nr_irqs));

	raw_spin_unlock_irqrestore(&pp->lock, flags);

	if (bit < 0)
		return -ENOSPC;

	for (i = 0; i < nr_irqs; i++)
		irq_domain_set_info(domain, virq + i, bit + i, pp->msi_irq_chip,
				    pp, handle_edge_irq, NULL, NULL);

	return 0;
}
```

另外，每个entry的doorbell地址必须和前面设置的 `PCIE_MSI_ADDR_LO` 和 `PCIE_MSI_ADDR_HI` 中的地址一致。

# 中断处理过程

同一个PCIe控制器中的MSI中断触发时，物理中断号都是相同的，软件需要根据 `PCIE_MSI_INTRn_STATUS` 中被拉高的bit来匹配中断处理函数。

物理中断触发后进入 `dw_handle_msi_irq` 函数，该函数找寻目前被拉高的bit，并找到对应的virq，匹配到对应的处理函数。

```c
/* MSI int handler */
irqreturn_t dw_handle_msi_irq(struct pcie_port *pp)
{
	printk("dw_handle_msi_irq\n");
	int i, pos, irq;
	unsigned long val;
	u32 status, num_ctrls;
	irqreturn_t ret = IRQ_NONE;

	num_ctrls = pp->num_vectors / MAX_MSI_IRQS_PER_CTRL;

	for (i = 0; i < num_ctrls; i++) {
		dw_pcie_rd_own_conf(pp,
				    PCIE_MSI_INTR0_STATUS +
					    (i * MSI_REG_CTRL_BLOCK_SIZE),
				    4, &status);
		if (!status)
			continue;
		printk("dwc-msi-handle: intr%d_status: 0x%x\n", i, status);

		ret = IRQ_HANDLED;
		val = status;
		pos = 0;
		while ((pos = find_next_bit(&val, MAX_MSI_IRQS_PER_CTRL,
					    pos)) != MAX_MSI_IRQS_PER_CTRL) {
			irq = irq_find_mapping(pp->irq_domain,
					       (i * MAX_MSI_IRQS_PER_CTRL) +
						       pos);
			printk("dwc-msi-handle: bit_pos: %d, virq: %d\n", pos, irq);
			generic_handle_irq(irq);
			pos++;
		}
	}

	return ret;
}
```

中断处理结束后还需要deactivated操作：

```c
static void dw_pci_bottom_ack(struct irq_data *d)
{
	
	struct pcie_port *pp = irq_data_get_irq_chip_data(d);
	unsigned int res, bit, ctrl;

	ctrl = d->hwirq / MAX_MSI_IRQS_PER_CTRL;
	res = ctrl * MSI_REG_CTRL_BLOCK_SIZE;
	bit = d->hwirq % MAX_MSI_IRQS_PER_CTRL;
	printk("dw_pci_bottom_ack: %d\n", bit);

	dw_pcie_wr_own_conf(pp, PCIE_MSI_INTR0_STATUS + res, 4, BIT(bit));
}
```

# 虚拟化

PCI-MSI的虚拟化的关键就在于将中断相关的寄存器的切分，以及如何将一个中断路由到不同的CPU。

## 统一的硬件doorbell

每个虚拟机都会对DW控制器的doorbell进行设置，hvisor需要设置影子寄存器，并在初始化时，自己生成一个doorbell写入硬件寄存器，使得在硬件上只能有一个doorbell。

同样，在虚拟机设置设备的MSI配置时，也需要对doorbell进行shadow，MSI CAP只需要在配置空间操作即可，MSIX则要求hvisor将设备MSIX table所在的BAR区域进行访问控制（包括了doorbell和msg），取代原先的直通方案。

## 寄存器切分

设置一个bit分配器，在进行zone的pci初始化时，记录该zone的PCI设备所需的MSI的总数，通过分配器分配一个硬件的起始bit。

例如，zone0需要10个MSI中断，此时bit分配器的下一个空闲bit为bit0，则zone0分配到的hwirq为0，此时下一个空闲bit为bit10；zone1需要8个MSI中断，则zone1分配到的hwirq为bit10。

那此时就存在一个虚拟机视角的 `virq_bit` 和实际虚拟机使用的硬件的 `hwirq_bit` 的映射。

以zone1为例：

- zone1打算为设备配置MSI msg，写入的bit是虚拟机视角的 `virq_bit`，例如为0，那么在hvisor的访问控制中，应该将该msg加上 `hwirq_bit` 的值，即变为10。
- zone1将设置 `ENABLE` 寄存器中的bit0为1，hvisor进行代理访问时，应该将硬件上的bit10设置为1，所以应该将虚拟机将要设置的值左移 `hwirq_bit`，并只保留低位8个bit的设置。同时从硬件读取原来的值进行组合，保证不会将其他虚拟机enable的中断关闭。
- 当属于zone1的设备触发中断时，`STATUS` 寄存器中的bit10被拉高，这个结果不能直接返回给zone1，zone1希望看到的是自己视角的bit0被拉高，所以需要将结果右移 `hwirq_bit` 位，并保留8个bit的结果（高位的结果可能是属于其他虚拟机的，可能不为0）。

## 中断路由

PCI-MSI在硬件上只有一个中断，而不同的虚拟机的pCPU不同，所以可以将硬件上的中断路由到CPU0，在中断处理函数中，读取 `STATUS` 寄存器的值，判断目前拉高的bit属于哪个虚拟机，将中断路由到该虚拟机的 `boot_Cpu`。

- 如果目标CPU是CPU0，则直接注入。
- 如果目标CPU是其他CPU，则发送一个IPI到其他CPU，在事件处理函数中直接注入中断。（全局设置每个CPU的待注入队列，以区别不同控制器的中断号）
